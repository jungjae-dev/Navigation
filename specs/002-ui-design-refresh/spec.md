# Feature Specification: 사용자 화면 디자인·레이아웃 통일 리팩토링

**Feature Branch**: `002-ui-design-refresh`

**Created**: 2026-06-07

**Status**: Draft

**Input**: User description: "현재 앱을 기능개발을 우선해서,, 디자인이 작업이 덜되어 있어. 주행화면을 제외한 모든 화면에서 레이아웃 및 디자인을 리팩토링 해서 깔끔한 디자인으로 개선하고 싶어. 애플 지도 참조해서,, 심플하고 통일성 있고, 깔끔한 디자인으로 개선해줘."

## Overview

기능 개발을 우선하다 보니 화면마다 디자인 완성도와 스타일이 제각각인 상태다. 주행(turn-by-turn) 화면을 제외한 모든 사용자 화면을, 애플 지도(Apple Maps)를 레퍼런스로 삼아 **심플하고, 통일성 있고, 깔끔한** 하나의 일관된 디자인 언어로 정리한다. 기능과 화면 흐름은 그대로 유지하되, 요소 배치·간격·계층은 재구성할 수 있다.

- Q: 홈/지도 화면의 플로팅 컨트롤을 어떻게? → A: 최소화·통합 (항목 재배치 포함)
- Q: 강조색(인디고) 사용 범위는? → A: 절제된 강조색 (주요 액션·선택·활성·링크에만 사용, 나머지는 중립색)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 모든 화면이 하나의 앱처럼 통일된 인상 (Priority: P1)

사용자가 홈/지도, 검색, 경로 미리보기, 장소·정류장 상세, 대중교통(버스 노선·시간표), 설정 화면을 오가는 동안, 색상·서체·간격·버튼·카드·시트 등 모든 시각 요소가 일관되어 "같은 앱의 같은 디자인"이라는 인상을 받는다. 어떤 화면에서도 톤이 튀거나 임시방편으로 만든 듯한 부분이 보이지 않는다.

**Why this priority**: 통일성은 이번 작업의 핵심 가치다. 화면별로 따로 손보는 것이 아니라 공통 디자인 토큰/컴포넌트를 단일 기준으로 정리하고 모든 화면이 이를 따르게 만드는 것이 MVP다. 이것만 완료되어도 앱 전체의 완성도가 즉시 올라간다.

**Independent Test**: 인스코프 화면들을 순서대로 캡처해 나란히 비교했을 때, 강조색·본문/제목 서체·여백 단위·모서리 곡률·버튼 스타일이 모든 화면에서 동일한 규칙을 따르는지 확인하면 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 인스코프 화면 어디에서나, **When** 강조색(액센트)이 쓰인 요소를 보면, **Then** 모든 화면에서 동일한 단일 강조색 체계를 사용한다(하드코딩된 임의 색 없음).
2. **Given** 두 개 이상의 다른 화면, **When** 본문/제목/캡션 텍스트를 비교하면, **Then** 동일한 서체 계층(타이포그래피 스케일)을 따른다.
3. **Given** 인스코프 화면의 버튼·카드·시트·목록 셀, **When** 같은 종류의 컴포넌트를 비교하면, **Then** 동일한 형태·여백·곡률·상태 표현을 갖는다.
4. **Given** 라이트/다크 모드 전환, **When** 인스코프 화면을 보면, **Then** 두 모드 모두에서 가독성과 통일성이 유지된다.

---

### User Story 2 - 애플 지도풍의 깔끔한 주요 화면 (Priority: P2)

사용자가 가장 자주 쓰는 홈/지도, 검색, 경로 미리보기 화면이 애플 지도처럼 군더더기 없이 정돈되어 보인다. 지도는 콘텐츠가 주인공이 되고, 하단 시트(드로어)·검색 바·플로팅 버튼 등은 최소한의 크롬으로 깔끔하게 떠 있으며, 정보 위계가 한눈에 들어온다.

**Why this priority**: 사용 빈도가 가장 높은 화면이라 체감 개선 효과가 크다. P1의 공통 디자인 언어를 가장 눈에 띄는 표면에 적용해, 애플 지도풍의 "심플하고 깔끔한" 목표를 직접적으로 실현한다.

**Independent Test**: 홈/지도·검색·경로 미리보기 화면을 애플 지도와 나란히 두고, 여백·계층·시트 형태·플로팅 컨트롤 정돈도를 비교해 "깔끔하다"는 기준을 충족하는지 확인하면 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 홈/지도 화면, **When** 화면을 보면, **Then** 지도 위 플로팅 컨트롤은 최소화·그룹화되어 정돈된 위치에 배치되고, 하단 시트는 그래버와 다단계 높이(detent)를 가진 불투명 솔리드 단일 시트로 표시된다.
2. **Given** 검색 화면, **When** 검색을 시작하거나 결과를 보면, **Then** 검색 바·최근 기록·자동완성·결과 목록이 일관된 간격과 셀 스타일로 정렬된다.
3. **Given** 경로 미리보기 화면, **When** 이동수단과 경로 옵션을 보면, **Then** 옵션 카드/셀·구분선·시작 버튼이 일관되고 위계가 분명하게 표시된다.

---

### User Story 3 - 상세·대중교통·설정 화면의 마감 일관성 (Priority: P3)

사용자가 장소 상세, 따릉이 대여소 상세, 버스 정류장 도착정보, 버스 노선 상세, 버스 시간표, 설정 화면에 진입했을 때도 주요 화면과 동일한 디자인 언어로 마감되어 있어, 깊은 화면까지 빈틈이 없다.

**Why this priority**: 상세·보조 화면은 사용 빈도는 낮지만, 통일성을 완성하려면 반드시 같은 기준으로 마감되어야 한다. P1/P2 이후 적용해도 가치가 누적된다.

**Independent Test**: 각 상세/보조 화면을 열어 헤더·섹션·목록 셀·구분선·버튼이 공통 컴포넌트 규칙을 따르는지, 임시 스타일이 남아있지 않은지 확인하면 단독 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 장소/대여소/정류장 상세 시트, **When** 헤더와 본문 콘텐츠를 보면, **Then** 공통 헤더·섹션·목록 스타일을 따른다.
2. **Given** 버스 노선 상세 및 시간표 화면, **When** 정류장 목록과 시간 정보를 보면, **Then** 통일된 서체 계층과 간격으로 표시된다.
3. **Given** 설정 화면, **When** 섹션과 항목을 보면, **Then** 일관된 그룹/행 스타일과 간격으로 정돈된다.

---

### Edge Cases

- **다크 모드**: 모든 인스코프 화면이 라이트/다크 두 모드에서 대비와 통일성을 유지하는가?
- **동적 글자 크기(Dynamic Type)/접근성**: 글자를 키웠을 때 레이아웃이 깨지지 않고 위계가 유지되는가?
- **긴 텍스트/한글·외국어**: 긴 장소명·주소·노선명이 잘림 처리 또는 줄바꿈으로 깔끔하게 처리되는가?
- **빈 상태(Empty State)**: 검색 결과 없음, 즐겨찾기/최근기록 없음, 도착정보 없음 등에서 일관된 빈 상태 표현을 제공하는가?
- **로딩 상태**: 데이터 로딩 중 표시가 일관된 스타일을 따르는가?
- **다양한 기기 크기**: 작은/큰 화면, 노치/다이내믹 아일랜드 환경에서 여백과 안전영역이 올바르게 유지되는가?
- **주행 화면과의 경계**: 주행 화면으로 진입/이탈할 때 이번 리팩토링이 주행 화면의 모양이나 동작을 바꾸지 않는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 색상·서체·간격·모서리 곡률·그림자·아이콘 크기에 대한 **단일 디자인 토큰 기준**을 가지며, 모든 인스코프 화면이 이 기준만을 따라야 한다.
- **FR-002**: 인스코프 화면은 화면별 하드코딩 색상을 사용하지 않고, 단일 강조색 체계를 포함한 공통 색상 토큰을 통해서만 색을 적용해야 한다.
- **FR-002a**: 강조색(인디고)은 **절제되게** 사용해야 한다 — 주요 액션(예: 시작/확인 버튼), 선택·활성 상태, 링크에만 적용하고, 그 외 표면·아이콘·텍스트는 중립색(레이블/회색 계층)을 사용한다.
- **FR-003**: 모든 텍스트는 공통 타이포그래피 계층(제목/본문/캡션 등)을 따라야 하며, 임의의 인라인 글자 크기·굵기 지정을 두지 않아야 한다.
- **FR-004**: 버튼·카드·하단 시트(드로어)·목록 셀·섹션 헤더·구분선·검색 바 등 반복되는 UI는 **공통 컴포넌트**로 통일되어 동일한 형태·여백·상태(기본/눌림/비활성/로딩)를 가져야 한다.
- **FR-005**: 간격과 여백은 공통 간격 단위 체계를 따라야 하며, 화면마다 다른 임의의 패딩 값을 사용하지 않아야 한다.
- **FR-006**: 시스템은 애플 지도를 레퍼런스로 한 **심플하고 깔끔한** 시각 스타일(콘텐츠 우선, 최소 크롬, 명확한 정보 위계, 정돈된 플로팅 컨트롤과 하단 시트)을 인스코프 화면에 적용해야 한다.
- **FR-006a**: 하단 시트(드로어)와 플로팅 요소의 표면은 **불투명 솔리드 배경**을 사용해야 한다(반투명 블러/머터리얼 사용 안 함). 대비와 가독성을 우선한다.
- **FR-007**: 기존 기능과 화면 흐름(어떤 화면에서 무엇을 하고 어디로 이동하는지)은 유지하되, 요소의 배치·간격·계층은 깔끔함을 위해 재구성할 수 있다.
- **FR-007a**: 화면마다 제각각인 드로어를, 그래버와 **다단계 높이(detent)** 를 가진 **크기 조절 가능한 단일 하단 시트 패턴**으로 통일해야 한다. 각 화면은 이 공통 시트 패턴 위에 콘텐츠를 얹는다.
- **FR-007b**: 홈/지도 화면의 상시 노출 플로팅 컨트롤(현재 위치·지도 모드·POI 레이어·따릉이 새로고침 등)은 애플 지도 수준으로 **최소화·그룹화**해야 하며, 사용 빈도가 낮은 항목은 메뉴 등으로 묶어 지도 가시 영역을 넓힌다(항목 재배치 허용).
- **FR-008**: 모든 인스코프 화면은 라이트/다크 모드 모두에서 일관성과 충분한 가독성(명도 대비)을 보장해야 한다.
- **FR-009**: 인스코프 화면은 동적 글자 크기 및 다양한 기기 크기·안전영역에서 레이아웃이 깨지지 않고 위계와 정돈감을 유지해야 한다.
- **FR-010**: 검색 결과 없음, 즐겨찾기/최근기록 없음, 도착정보 없음 등 **빈 상태와 로딩 상태**는 일관된 공통 스타일로 표현되어야 한다.
- **FR-011**: 본 리팩토링은 **주행(turn-by-turn) 화면**의 모양·레이아웃·동작을 변경하지 않아야 한다(주행 화면은 별도의 글랜스 최적화 디자인 유지).
- **FR-012**: 본 리팩토링은 **개발자 도구(DevTools)·녹화 파일 목록·CarPlay 화면**을 범위에서 제외한다(이번 작업의 통일 기준 적용 대상 아님).
- **FR-013**: 리팩토링 후 인스코프 화면에는 임시방편 스타일(임의 색/폰트/간격, 미완성 마감)이 남아있지 않아야 한다.

### In-Scope Screens

- 홈/지도 화면(지도, 플로팅 컨트롤, 홈 드로어: 즐겨찾기·카테고리·최근 검색)
- 검색 화면 및 검색 결과 화면
- 경로 미리보기/선택 화면
- 장소·따릉이 대여소·버스 정류장 상세 시트
- 대중교통: 버스 노선 상세, 버스 정류장 시간표
- 설정 화면(차량 모델 가져오기 등 설정 내 보조 화면 포함)
- 위 화면들이 공유하는 모든 드로어 컨테이너·셀·헤더·버튼 등 공통 컴포넌트

### Out-of-Scope Screens

- 주행(turn-by-turn) 화면 및 그 부속 뷰(턴 안내 배너, 하단 주행 바, 속도계, 도착 팝업, 리센터 버튼, 가상주행 컨트롤)
- 개발자 도구(DevTools), 녹화 파일 목록 화면
- CarPlay 화면

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 인스코프 화면 100%가 단일 디자인 토큰 기준(색상·서체·간격·곡률)을 따르며, 화면 고유의 하드코딩 스타일이 0건이다.
- **SC-002**: 같은 종류의 UI 요소(버튼/카드/시트/목록 셀/헤더)는 인스코프 전 화면에서 100% 동일한 공통 컴포넌트 규칙을 따른다.
- **SC-003**: 인스코프 화면 전부가 라이트/다크 모드 양쪽에서 본문 텍스트 명도 대비 WCAG AA(4.5:1) 이상을 만족한다.
- **SC-004**: 디자인을 모르는 제3자가 인스코프 화면 캡처를 비교했을 때 "하나의 통일된 깔끔한 앱"으로 인지한다(검토자 정성 평가에서 통일성 합격).
- **SC-005**: 주행 화면·DevTools·CarPlay의 모양과 동작에 변화가 없다(회귀 0건).
- **SC-006**: 동적 글자 크기를 기본 대비 한 단계 이상 키워도 인스코프 화면에서 레이아웃 깨짐(겹침·잘림으로 정보 손실)이 0건이다.
- **SC-007**: 정의된 빈 상태·로딩 상태가 해당되는 모든 인스코프 화면에서 일관된 스타일로 제공된다.
- **SC-008**: 인스코프의 모든 하단 시트가 그래버 + 다단계 detent를 가진 동일한 단일 시트 패턴을 사용하며, 화면별 개별 드로어 구현이 0건이다.
- **SC-009**: 강조색(인디고)은 주요 액션·선택·활성·링크 외의 표면/텍스트/아이콘에 사용되지 않는다(절제 기준 위반 0건).

## Assumptions

- "모든 화면"의 범위는 **사용자 대면 화면**으로 한정한다(사용자 확인 완료). DevTools와 CarPlay는 제외하며, CarPlay는 Apple CarPlay 템플릿 제약상 별도 과제로 둔다.
- 디자인 리팩토링은 **레이아웃 재배치를 허용**하되 기존 기능과 화면 간 이동 흐름은 보존한다(사용자 확인 완료). 정보 구조(IA)의 전면 재설계는 범위가 아니다.
- 레퍼런스인 "애플 지도"는 시각적 방향성(콘텐츠 우선, 최소 크롬, 둥근 하단 시트와 그래버, 정돈된 플로팅 컨트롤, 명확한 위계)을 의미하며, 픽셀 단위 복제가 아니다.
- 강조색은 기존 인디고 계열 액센트 체계를 기준으로 유지·정돈한다(메모리상 #4F46E5 / 다크 #818CF8). 색은 항상 공통 팔레트를 경유한다.
- 기존에 정의된 공통 디자인 토큰/컴포넌트 자산을 단일 기준으로 확장·정리해 재사용하며, 화면별 임시 스타일을 이 기준으로 흡수한다.
- 빈 상태·로딩·에러 표현은 도메인 표준의 사용자 친화적 기본값을 따른다.
- 본 작업은 시각·레이아웃 품질에 초점을 두며, 새로운 기능 추가나 데이터/비즈니스 로직 변경은 포함하지 않는다.
